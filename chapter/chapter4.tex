\section{Deployment Scenarios and Business Use Cases}

\subsection{SAML: Traditional B2B Federation and Legacy Architectures}
SAML 2.0 has been the de facto standard for enterprise Single Sign-On (SSO) and B2B federation for over two decades. Its XML-based assertions and robust security features made it ideal for large organizations with complex identity needs. However, its rigidity and verbosity can be a drawback in modern, agile environments. SAML is predominantly found in scenarios where organizations have deeply established trust relationships and require strict security guarantees, such as in government frameworks, higher education networks, and large-scale legacy enterprises.

\subsection{OAuth 2.0 \& OIDC: APIs, Mobile Apps, and Modern Ecosystems}
From an enterprise strategy perspective, transitioning to OAuth 2.0 and OpenID Connect (OIDC) is driven by the need to support lightweight, decentralized, and API-centric environments. Unlike the heavy SOAP/XML structures of SAML, OIDC leverages JSON and standard HTTP mechanisms. This significantly reduces network overhead and allows for stateless session management via JSON Web Tokens (JWTs), which is crucial for scaling cloud-native microservices.

Consequently, OAuth 2.0 and OIDC have become the standards of choice for modern applications, especially in the context of RESTful APIs, native mobile applications, and Single Page Applications (SPAs). OAuth 2.0's delegation model allows users to grant limited access to their resources without sharing credentials, making it ideal for third-party integrations. OIDC builds on top of OAuth 2.0 by adding a dedicated identity layer, enabling seamless authentication and standardized user profile management. These protocols are particularly well-suited for consumer-facing applications where user friction must be minimized.

\subsection{Hybridization Strategies: Transition and Proxying Patterns (SAML-to-OIDC)}
Many organizations face the complex challenge of transitioning from legacy SAML-based identity providers to modern OIDC architectures without interrupting daily operations. Hybridization strategies, such as deploying an Identity Provider (IdP) Proxy acting as a SAML-to-OIDC bridge, allow organizations to maintain existing corporate SAML integrations while gradually adopting OIDC for new application deployments. This architecture minimizes business disruption and enables a phased migration strategy. 

In our laboratory environment, we validated the functionality of a SAML-to-OIDC proxy, demonstrating that it is effectively possible translate incoming SAML assertions into compliant OIDC identity tokens, thereby enabling seamless interoperability between the two protocol generations (see Figure \ref{fig:saml_mock_architecture}).

\subsection{SSO Flows: Transparency and Perceived Friction (SAML vs OIDC)}
From a user experience perspective, the perceived friction of authentication flows can impact user adoption and satisfaction. SAML 2.0 operates on an implicit corporate trust model designed for enterprise intranets. Because trust is established statically by administrators exchanging metadata beforehand, the protocol skips user-facing consent steps entirely; the user authenticates and is instantly redirected back to the application. However, the lack of structural user prompts in SAML is offset by severe technical friction during session termination. Our laboratory testing exposed the fragility of the browser-mediated SAML Single Logout (SLO) mechanism. When a logout was triggered, the user's browser was caught in a series of synchronous, sequential redirect loops to handle session invalidation across domains:

\begin{verbatim}
1. GET http://localhost:88/logout
2. GET http://localhost:8080/realms/myrealm/protocol/saml?SAMLRequest=jVFdS8...
3. GET http://localhost:88/saml/sls?SAMLResponse=jVJNa8Mw...
4. GET http://localhost:88/login
\end{verbatim}

Because SAML SLO forces the browser to sequentially hit the Identity Provider and every active Service Provider via frontend redirects, any network latency or failure in a single downstream application causes the user interface to hang on a blank screen. Conversely, OIDC isolates the user experience from backend latency by utilizing asynchronous Back-Channel logouts. The user is instantly met with a responsive frontend redirect to the login page while server-to-server HTTP POST requests invalidate background sessions in silence.

\subsection{Empirical Performance Analysis: Token Size Impact (XML vs JSON)}
In our laboratory tests, we observed that the average size of SAML assertions was approximately 2 KB, while OIDC ID tokens averaged around 0.5 KB. This significant reduction in token size directly correlated with improved load times, with OIDC flows exhibiting an average latency reduction of 40\% compared to SAML. 

This stark contrast in data density is rooted in the protocol formatting requirements. Deconstructing a raw, intercepted SAML \texttt{LogoutRequest} reveals the high syntax-to-data ratio mandatory for XML-based enterprise schemas:

\begin{lstlisting}[language=XML, caption={Decoded SAML 2.0 Logout Request Payload}]
<samlp:LogoutRequest
  xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
  xmlns:saml="urn:oasis:names:tc:SAML:2.0:assertion"
  ID="ONELOGIN_eb1025f498469ecf11562b6f3c4ad31c1eba6c70"
  Version="2.0"
  IssueInstant="2026-05-24T14:25:24Z"
  Destination="http://localhost:8080/realms/myrealm/protocol/saml">
    <saml:Issuer>flask-client</saml:Issuer>
    <saml:NameID Format="urn:oasis:names:tc:SAML:2.0:nameid-format:transient">
      G-a244d68c-af01-4c8b-b32d-2ad4db0e38c7
    </saml:NameID>
</samlp:LogoutRequest>
\end{lstlisting}

Because SAML relies on rigid, deeply nested XML namespaces (\texttt{xmlns:samlp}), cryptographic tracking states (\texttt{ID}), and explicit formatting wrappers just to transmit a basic command, the payload size balloons. To pass this via an HTTP-Redirect binding, the client application must programmatically compress this XML tree and transform it into a URL-encoded Base64 string. This places a continuous encoding and decoding computational tax on the web client.

OIDC drastically minimizes this footprint by operating over HTTP query parameters and compact JSON Web Tokens (JWTs). To execute an identical session termination, OIDC avoids heavy XML structures entirely, mapping the request to a highly optimized, human-readable URL query:

\begin{verbatim}
https://localhost:8080/.../logout?id_token_hint=eyJhbGciOi...&client_id=flask-client
\end{verbatim}

Instead of managing stateful transactions via transient IDs, OIDC clients echo the stateless, compact JWT token they already hold. This performance optimization becomes critical in mobile environments with restricted bandwidth, where OIDC's lightweight footprint translates into a fluid and responsive user experience.

\subsection{RGPD: Data Minimization and User Consent}

An essential criterion for enterprise protocol selection is alignment with modern data protection frameworks, most notably the European Union's General Data Protection Regulation (GDPR). Central to this regulation is the principle of "data minimization" outlined in Article 5(1)(c), which mandates that personal data must be adequate, relevant, and limited to what is necessary in relation to the purposes for which they are processed~\cite{gdpr_2016}. Achieving this requirement natively depends a lot on how each protocol handles identity attributes.

SAML was designed in an era prior to strict privacy regulations. By default, its Core Specification relies on an assertion structure where a pre-configured set of user attributes (\texttt{AttributeStatement}) is packaged and signed as a single block sent to the SP~\cite{oasissaml}. However, the protocol can be configured to include only required attributes desired by a SP on the IdP side, but this is not a native feature of the protocol and requires additional administrative overhead to maintain attribute distribution policies.

Conversely, OpenID Connect was built with modern privacy constraints in mind. According to the OIDC Core Specification, attribute divulgation is dynamically governed by the use of specific OAuth 2.0 \textit{scopes} (such as \texttt{openid}, \texttt{profile}, or \texttt{email}) and can be retrieved via the user info endpoint~\cite{oidc_core}.

Our laboratory experiments substantiated this architectural divergence. When analyzing the authentication traffic via network interception, we observed that the SAML Identity Provider systematically transmitted the entire configured array of user attributes within the XML assertion, regardless of the specific client application's immediate functional needs. This pattern poses a distinct compliance challenge regarding GDPR's data minimization mandate. 

In contrast, our OIDC implementation successfully validated that the Identity Token (\texttt{id\_token}) and the \texttt{/userinfo} response restricted user data to the claims requested by the specific scope authorized by the user. This demonstrates that OIDC provides a much more robust, native technical mechanism for enforcing \textit{Privacy by Design}.



\subsection{Financial Sector: Payment Service Directive (PSD) and Financial-grade API (FAPI)}
The PSD in the European Union mandates banks to open up customer account data to third-party providers via secure communication interfaces, requiring strict Strong Customer Authentication (SCA)~\cite[30, article 97]{psd2_2015}. Because SAML 2.0 relies on front-channel browser redirections and heavy XML payloads, it is fundamentally unsuited for modern API-driven banking ecosystems which increasingly target mobile platforms.

While core OAuth 2.0 handles API authorization, its baseline security model is insufficient for high-value financial transactions due to vulnerabilities like token leakage and authorization code injection. Consequently, the OpenID Foundation developed the \textit{Financial-grade API} (FAPI) framework on top of OIDC and OAuth 2.0~\cite{fapi_1_core}. FAPI hardens OIDC by enforcing advanced security profiles (e.g. mutual TLS (mTLS) or Private Key JWT for client authentication, etc) making the OIDC ecosystem the global standard for Open Banking compliance.
